分布式系统架构设计

发表于 2020-04-08

1. 为什么要有分布式系统？

换句话说：分布式系统，在解决什么问题？

主要解决2个问题：增大系统容量 和 加强系统可用性

  • 增大系统容量
    • 业务量越来越大，单机无法支撑
    • 此时，需要垂直拆分 或 水平拆分，变成分布式系统
  • 加强系统可用性
    • 业务越来越重要，要让故障的时间尽可能少
    • 不能因为一台机器故障，导致整体不可用
    • 架构中不能存在单点故障，提高系统的可用性

名词解释

  • SOA
    • 全称：Service-oriented Architecture
    • 中文：面向服务架构
    • 变迁：从SOAP、WSDL –> Restful + JSON，即传统SOA向 微服务的转变
    • 传统SOA：服务间走网络接口，RPC协议，数据库共用，
    • 微服务：HTTP协议，按服务拆分数据库
  • ESB
    • 全称：Enterprise Service Bus
    • 中文：企业服务总线
    • 变迁：被后来的MQ所取代

2. 分布式系统中的问题

  • 异构系统的不标准问题
    • 软件和应用不标准
    • 通讯协议不标准
    • 数据格式不标准
    • 开发和运维 的 过程和方法 不标准
  • 系统架构中，服务依赖的问题
    • 按重要性拆分，防止重要的依赖非重要的
    • 数据库按照业务拆分
  • 故障发生的概率更大
    • 面向故障设计，关键点监控
    • 人管代码、代码管机器、人不管机器
  • 多层架构运维的复杂度更大
    • 基础层：机器、网络、存储
    • 平台层：Tomcat、MySQL、Redis、Kafka等
    • 应用层：各种功能服务
    • 接入层：网关、负载均衡、CDN、DNS等

3. 分布式技术栈

3.1 做好2件事

  • 大流量处理
    • 大规模请求，分发到不同的机器上
  • 关键业务保护
    • 把故障隔离起来，防止雪崩
    • 如果流量过大，需要对业务降级，保护关键业务流程

3.2 提高架构性能

  • 缓存
    • 提高系统的访问能力最有效的手段
    • 一个缓存集群，需要一个Proxy来做分片和路由
  • 负载均衡
    • 多机器分担同一个服务的压力
  • 异步调用
    • 让请求排队，把请求的峰值削平，后端以自己的速度来处理
    • 可以增加系统的吞吐量，但实时性会差很多
    • 为了防止消息丢失，还要将消息持久化，并存储状态，增加了复杂度
  • 数据镜像
    • 在任意一个点上写，内部自动同步好
    • 查询简单，可能会存在数据一致性的问题
  • 数据分区
    • 数据按照一定的方式分区，如地理位置
    • 不同的数据区，分担不同区的流量
    • 需要数据路由中间件，导致跨库的join 和 跨库的事物复杂
    • 初期、读写分离，后期分库分表

3.3 提高架构稳定性

  • 服务拆分
    • 两个目的：隔离故障、重用服务模块
    • 会引入服务间调用的依赖
  • 服务冗余
    • 去除单点故障
    • 支持服务的弹性伸缩 和 故障迁移
  • 限流降级
    • 一种保护措施，确保整个架构不会挂掉
    • 停掉一部分服务、或 限制一部分用户
  • 高可用架构
    • 从冗余架构角度，来保障可用性
    • 多租户隔离、灾备多活等
  • 高可用运维
    • DevOps中的CI/CD
    • 自动化测试、灰度发布、线上系统的自动化控制
    • 让「计划内」和「计划外」的宕机时间最短

3.4 分布式系统的关键技术

  • 服务治理
    • 服务拆分、服务调用、服务发现、服务依赖、服务的关键度定义
    • 梳理出，服务间的依赖关系 和 调用链
  • 架构软件管理
    • 架构版本
  • DevOps
  • 自动化运维
    • 自动伸缩、故障迁移、配置管理、状态管理等
  • 资源高度管理
    • 云计算IaaS层的计算、存储、网络等资源的高度、隔离 和 管理
  • 整体架构监控
    • 对应用层、中间层、基础层进行监控
  • 流量控制
    • 负载均衡、服务路由、熔断、降级、限流、灰度发布等

3.5 分布式系统的「纲」

  • 全栈系统监控
  • 服务/资源 调度
  • 流量调度
  • 状态/数据调度
  • 开发 和 运维 的自动化

3.5.1 全栈系统监控

  • 多层体系的监控
    • 基础层：CPU、内存、网络吞吐、硬盘IO、硬盘使用
    • 中间层：Nginx、Redis、ActiveMQ、Kafka、MySQL、Tomcat等
    • 应用层：HTTP访问的吞吐量、响应时间、返回码、调用链路、性能瓶颈等
  • 好的日志系统
    • 关注整体应用的SLA：从提供服务的API来监控
    • 关联指标聚合：有关联的系统、指标聚合展示
    • 快速故障定位：用户请求跟踪的trace监控，即请求在分布式系统中的调用链
    • 两个主要场景
      • 体检
        • 容量管理：机器、资源
        • 性能管理：找到系统瓶颈，有针对性的优化系统
      • 急诊
        • 定位问题：暴露并找到问题的发生点
        • 性能分析：快速找到系统的瓶颈
  • 如何做出一个好的监控系统？
    • 服务调用链跟踪
      • 从对外的API开始
      • 将后台的实际服务给关联起来
      • 再进一步将这个服务的依赖服务关联起来
      • 直到最后一个服务，如MySQL、Redis
      • 最佳实践：Google Dapper，开源的Zipkin
      • 可以通过Java字节码注入，做到代码无侵入
    • 服务调用时长分布
      • 知道最耗时的服务是哪个
    • 服务的TOP N视图
      • 按调用量排名
      • 按请求最耗时排名
      • 按热点排名
    • 数据库操作关联
      • 通过JavaAgent字节码注入技术，拿到JDBC执行数据库操作的执行时间
    • 服务资源跟踪
      • 服务与机器节点上的数据（CPU、内存、IO、Disk、Network）关联起来

3.5.2 服务调度

  • 面向对象设计原则solid
    • 单一职责
    • 开闭
    • 里氏替换
    • 接口隔离，多个小接口，而不是一个大接口
    • 依赖反转
    • 迪米特，最少知道
    • 组合优于继承

  • 集群内节点共享数据

    • Paxos
    • Raft
    • Gossip

  • 分布式数据存储3大问题

    • 数据副本
      • Master-Slave
      • Master-Master
      • 两阶段 和 三阶段 提交
      • Paxos
    • 数据一致性
    • 分布式事务

  • 分布式事务

    • 应用层：两阶段提交
    • 数据库层：Paxos

  • 分布式文件系统

  • 弹力设计（容错能力），保障可用性
    • 认知故障
      • 可测量，可用性 = 平均故障前时间 / （平均故障前时间 + 平均故障恢复时间）
    • 隔离
      • 按服务隔离
      • 按用户请求隔离（租户）
    • 异步通讯
      • 请求响应式，发了请求，轮询结果
      • 观察者模式，有了事件，同步调用
      • MQ模式，事件仍给MQ中间件
    • 幂等，一个订单，多次支付请求，是否会多次支付？
      • 产生UUID，雪花算法
      • 是没有副作用，不是结果相同
    • 服务的状态
    • 补偿事务
      • 事务
        • ACID
          • 原子、一致、隔离、持久
        • BASE
          • 基本可用、软状态、最终一致
      • 补偿机制
        • 努力的把一个流程执行完
        • 如果执行不下去，需要启动补偿机制，回滚业务流程
    • 重试
      • 暂时的故障，重试，可能解决
    • 熔断Hystrix
      • 闭合、打开、半开
    • 限流
      • 计数器
      • 队列
      • 漏斗算法
      • 令牌桶算法
      • 基于响应时间的
    • 降级
      • 降低一致性：强一致性 降为 最终一致性
      • 停止次要功能
      • 简化功能
  • 管理设计
    • 分布式锁
      • 安全性（排他性）
      • 避免死锁
      • 容错性
      • 是否需要？
        • CAS是否能解决？
        • 进程间的同步、互斥？
    • 配置中心
      • 静态配置
        • 服务启动时确定的，不会改的
      • 动态配置
        • 按运行环境分
          • 开发、测试、预发、生产等
        • 按依赖区分
          • 外部，如连接地址
          • 内部，如日志级别、降级开关
        • 按层次分
          • 基础层，如操作系统
          • 中间层，如tomcat
          • 应用层，应用自己的配置
    • 边车模式
      • 逻辑与控制分离
        • 服务，实现好自己的逻辑
        • 控制，交给边车处理，如 监视、日志、限流、熔断、服务注册、协议适配转换等
    • 服务网格
      • 把边车管理起来的东西
    • 网关模式
    • 部署升级策略
      • 停机
      • 蓝绿
      • 滚动
      • 灰度
      • AB
  • 性能设计
    • 缓存
      • Cache Aside更新模式
        • 读完放缓存
        • 写入时，让缓存失效
      • Read/Write Through更新模式
        • 通过缓存读写数据库
        • 上层只用管一个操作即可
        • 缓存当作存储自带的东西
      • Write Behind Caching更新模式
        • 只写内存
    • 异步处理
    • 数据库扩展
    • 秒杀
    • 边缘计算
